Code review — triage order

  1. Correctness — logic, off-by-one, null, races
  2. Concurrency / resources — locks, leaks, OOM
  3. Security — injection, pickle, secrets
  4. Reliability — retries, timeouts, backpressure
  5. Observability — metrics not print, HTTP codes
  6. Style / naming — last, not first

Google Doc rules

  • Manual trace + propose test cases
  • Narrate aloud; ask when stuck
  • 200-line CL: understand before fixing
  • Style nits are often decoys

Primitives

  • Lock = exclusive access
  • Semaphore(N) = cap concurrency at N
  • Always release in finally / context mgr

NALSD / Systems Architecture

Not greenfield — fix existing system

  • Clarify scope, scale, constraints first (~20 min)
  • BOM: machines, cores, disk, bandwidth
  • Iterate: possible → feasible → resilient
  • Incremental ops fixes > re-architect

When overloaded

  • Admission control · load shed · bounded queues
  • Return 503 early vs deep queue timeout
  • Fix tail latency (p99), not just average

Four golden signals

  • Latency · Traffic · Errors · Saturation

SLO / error budget

  • Agree SLIs before boxes
  • Exhausted budget → negotiate with data

Physics traps — say impossible = pass

  • 5 PB / 10 Gbps ≈ weeks, not 4h RTO
  • 1 TB → 10k nodes: tree/P2P fanout, not serial
  • 78% avg CPU → bursts → queue → timeouts at peak

Troubleshooting — first moves

  • Impact — who, how much, since when?
  • Changes — deploys, flags, config, traffic
  • Mitigate before root cause (one change at a time)
  • Golden signals — don't trust CPU alone

Shakespeare blackbox alert

  • Probe red ≠ always user impact — verify real traffic
  • Check pipeline: index → serve → deps

Synthetic green, users red

  • Probe misses path / region / cohort
  • Compare canary coverage vs prod traffic

Post-deploy latency spike

  • Correlate timeline · canary vs full · rollback candidate

Googleyness & Leadership (C3, C5)

  • STAR: Situation → Task → Action → Result (+ Learning)
  • Prep: conflict with manager, critical feedback, teammate disagreement
  • 25 min G&L can be fine — conversational rapport scores
  • High vs low performer: differentiated coaching, not identical treatment
  • Error budget exhausted: negotiate with data, don't just veto
  • Burnout: alert hygiene, fair rotation, reduce toil — not heroics
  • Sev-1 during demo: IC role, calm comms, mitigate first
  • Four attributes: GCA · RRK · Leadership · Googleyness

IC coding awareness (not your loop, but reported)

  • tail large file: seek backward in chunks
  • Word frequency: stream once, hash map count
  • Confusing cities: count (city, name) pairs ≥ 3
  • Deadlock vs livelock: blocked forever vs busy, no progress

Systems Architecture — 60 min flow

0–20 min: Clarify — users, QPS, read/write ratio, latency, retention, consistency, failure tolerance
Then: High-level boxes + data flows (API, storage, cache, queue, CDN)
Deep-dive when asked: One component — APIs, keys, failure behavior
Always volunteer: How you know it's healthy (lag, drops, SLO alerts)
Vague prompts ("trends in unstructured data"): clarify data volume, consumers, latency, privacy, what "trend" means
Toy prompts ("snakes game"): still do requirements + math + SLOs + failure modes
Interviewer disengaged in NALSD: likely missed early clarifier — pause and restate scope
