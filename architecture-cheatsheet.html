Canonical NALSD architectures (official Google exercises)

AdWords CTR (Workbook Ch.12)

query log ─┬→ QueryStore (by query_id) click log ─┴→ LogJoiner → ClickMap ─┐ query log ───→ QueryMap ────────────┴→ CTR dashboard
  • 500K queries/s · 10K clicks/s · ~2 KB/query
  • ~100 TB/day query logs · <5 min freshness SLO
  • Shard LogJoiner by query_id; QueryMap/ClickMap by ad_id
  • MapReduce rejected: tiny batches break join correctness
  • Multi-DC: Paxos replicas; ~25 ms/op → parallelize writes

Distributed ImageServer (SRE Classroom)

Client → HTTP LB → Upload/Download/Search svc ↓ ↓ Metadata/Index svc → Blob storage (S3-like) Search → thumbnails from storage + CDN
  • 1M users · 50 img/day · 8 KB metadata/img
  • ~0.5 TB/day metadata · 30-day retention ≈ 15 TB
  • Split read/write paths (uploads hold connections longer)
  • Search index: global consistency caps write QPS (~6 at 150 ms RTT)
  • SLOs: upload <250 ms · download <200 ms · search <250 ms (p99.9)

Distributed PubSub (SRE Classroom)

push(topic,msg) → router → sharded message store ↓ (local + cross-DC replicas) pop(topic) ← subscriber ← any DC within 1s
  • 5K topics · 10K msg/day/topic · 50M msg/day
  • 100-day retention · ~25 TB total store
  • At-least-once · lose ≤0.01%/yr (~5K msg/day budget)
  • Shard into buckets; replicate locally + cross-DC
  • Sync vs async replication: durability vs ~10 ms push latency

Why not MapReduce alone?

  • Small non-overlapping batches: query in batch 1, click in batch 2 → never joined
  • 5-min freshness SLO forces streaming or micro-batch with overlap

Latency order (fast → slow)

  • CPU cache / RAM → SSD → HDD → cross-region RTT
  • Dashboard in <1s: data in RAM/index, not WAN per row
  • Wrong shard key = hot partition — same math, different bottleneck
